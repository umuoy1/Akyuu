GitHub 监控聚合 Agent  
最终态产品里程碑路线图

面向“完整产品终局”而非 MVP 的推进拆解｜用于后续按里程碑继续细分任务、排期与组织研发

**终局定义：**这不是一个只会抓事件、生成日报的工具，而是一个面向个人与团队的 GitHub 技术情报平台。

**最终态应同时具备：**多对象监控、跨仓库主题理解、趋势雷达、历史可追问、个性化排序、协作消费、行动闭环、开放集成，以及企业级治理与可扩展性。

# 一、路线图设计原则

• 每个里程碑代表一个“新的产品状态”，而不是零散功能的堆积。

• 里程碑顺序遵循：先把情报底座做对，再把产品闭环做成，再把个性化、协作、平台化逐层叠加。

• 任何下一阶段拆分，都应以“跨到下一个产品状态所需的最小必要工作”为准，而不是简单列功能清单。

# M0｜终局定义冻结

|     |     |
| --- | --- |
| **目标状态** | 把“完整产品终局”定义清楚，并冻结核心边界、价值主张、用户对象、竞争位置与成功标准。 |
| **必须建成的能力** | • 明确产品终局：个人与团队共用的 GitHub 技术情报平台，而不是单纯日报器。  <br>• 定义目标用户层级：重度开源观察者、技术方向跟踪者、团队负责人、研究/分析型用户。  <br>• 定义完整产品能力树：Watch、Repo Intelligence、Topic Intelligence、Trend Intelligence、History、Ask、Personalization、Collaboration、Actions、Platform、Enterprise。  <br>• 定义不做的事情：不做代码托管替代、不做泛化 DevOps 平台、不做通用 BI。 |
| **阶段产出** | • 正式版 PRD / 产品地图 / 名词表 / 终局定义文档  <br>• 长期里程碑路线图  <br>• 情报对象模型的顶层抽象（RepoWatch / TopicWatch / TrendWatch / TeamSpace 等） |
| **退出标准** | 团队内部对“这是什么产品、最后会长成什么样”不再存在根本分歧；后续讨论开始围绕阶段推进，而不是反复争论方向。 |

# M1｜情报数据底座

|     |     |
| --- | --- |
| **目标状态** | 把公开 GitHub 生态中的关键信号源稳定接入，形成可持续采集、可追溯、可重放的情报数据底座。 |
| **必须建成的能力** | • 采集公开仓库与组织的基础事件流、PR / Issue / Release / Commit 详情。  <br>• 接入 Trending 日榜快照与 github-weekly-rank 这类外部榜单源。  <br>• 建立调度、重试、去重、失败告警、速率限制与配额控制。  <br>• 建立原始信号存储、快照存储与观测面板。 |
| **阶段产出** | • Collector 服务、调度器、原始数据表、快照表、任务日志  <br>• 一套长期可运行的数据同步策略  <br>• 内部可见的采集监控页面或日志体系 |
| **退出标准** | 系统可以持续运行而不是一次性抓取；任何上层功能都不再依赖手工导入数据。 |

# M2｜统一事件与知识模型

|     |     |
| --- | --- |
| **目标状态** | 把杂乱原始信号翻译成统一的事件语言、实体模型与主题知识图谱，让后续能力建立在统一理解层上。 |
| **必须建成的能力** | • 定义 Canonical Event：如 PR Opened、PR Merged、Issue Became Hot、Repo Entered Trending、Proposal Discussed 等。  <br>• 定义实体层：Repo、Org、Author、Proposal、Topic、TrendItem、TeamSpace。  <br>• 定义 Topic Graph：别名、关键词、关联仓库、关联路径、关联人物、证据规则。  <br>• 建立初版去重、合并、时序压缩与证据关联机制。 |
| **阶段产出** | • 统一 schema 与 normalizer  <br>• Entity Resolver / Topic Resolver  <br>• 知识图谱或等价的数据结构 |
| **退出标准** | 下游摘要、推荐、趋势分析与问答都不再直接依赖原始 payload，而是建立在统一事件与知识对象之上。 |

# M3｜Repo Intelligence 产品化

|     |     |
| --- | --- |
| **目标状态** | 让围绕仓库的情报消费能力独立成立：用户已经可以不刷 GitHub，而通过产品读懂关注仓库的重点变化。 |
| **必须建成的能力** | • 对单仓与多仓生成高质量日摘要、周摘要与推荐阅读。  <br>• 建立重要性排序：事件权重、影响范围、讨论热度、用户优先级。  <br>• 识别并压缩噪声：重复同步、机器人变更、文档 typo、批量格式化更新等。  <br>• 对高价值 PR / Issue / Release 提供摘要与推荐理由。 |
| **阶段产出** | • Repo Digest Pipeline  <br>• 重要性打分 v1 与推荐阅读模块  <br>• Today 页面中围绕 RepoWatch 的核心摘要区块 |
| **退出标准** | 对于只关注仓库的人群，产品已经具备第一层独立价值：能替代手工刷 watchlist 的大部分成本。 |

# M4｜Topic Intelligence 产品化

|     |     |
| --- | --- |
| **目标状态** | 将产品核心差异化能力做出来：它不仅知道某个仓库发生了什么，还能回答某个主题、提案、方向是否在推进。 |
| **必须建成的能力** | • 支持跨仓库 TopicWatch 的配置与管理。  <br>• 对 TC39 这类场景进行主题归并：proposal repo、notes、agendas、ecma262、test262 等进入同一 Topic Timeline。  <br>• 输出主题级结论：讨论推进、规范推进、测试推进、阶段变化候选信号。  <br>• 形成主题详情页与主题证据链视图。 |
| **阶段产出** | • TopicWatch 产品形态  <br>• Topic Timeline 与 Topic Digest  <br>• 主题级信号解释与证据跳转 |
| **退出标准** | 用户第一次能从系统得到真正的“议题级判断”，产品开始明显区别于传统监控工具。 |

# M5｜Trend Intelligence 产品化

|     |     |
| --- | --- |
| **目标状态** | 把 repo / topic 之外的宏观生态趋势纳入同一个情报框架，形成日榜、周榜与结构性变化的解释层。 |
| **必须建成的能力** | • Trending 日榜快照与昨日 diff 常态化。  <br>• 周榜接入与日榜交叉判断，区分短时热点与持续趋势。  <br>• 对热门 repo 聚类，输出方向性结论，例如 Agent、Infra、AI DevTool 等结构变化。  <br>• 为用户关注对象建立“微观变化”与“宏观背景”的关联。 |
| **阶段产出** | • TrendWatch 产品页  <br>• Trend Diff Engine 与聚类模块  <br>• 宏观趋势叙事能力 |
| **退出标准** | 产品不再只回答“我关注的对象发生了什么”，还可以回答“整个生态最近在往哪里动”。 |

# M6｜报告与历史闭环

|     |     |
| --- | --- |
| **目标状态** | 把情报能力真正包装成产品闭环：从 watch 配置，到日报/周报生成，再到历史回看、检索与多渠道交付完整跑通。 |
| **必须建成的能力** | • 支持 Today / Watches / Topics / Trends / History 等核心界面。  <br>• 支持日报、周报、月度回顾等多周期报告。  <br>• 支持邮件等交付方式，让用户不用进入产品也能消费摘要。  <br>• 支持历史回看、搜索、过滤与跨时间窗口对比。 |
| **阶段产出** | • 完整的 Web 端产品骨架  <br>• 多周期报告生成器  <br>• History 与检索体系 |
| **退出标准** | 一个新用户可以独立完成配置并连续消费产物；产品从后台系统成长为真正可用的前台体验。 |

# M7｜Askable Analyst 成型

|     |     |
| --- | --- |
| **目标状态** | 让产品从阅读型情报系统升级为对话型分析助手：用户不仅能看，还能继续追问、比较、展开与验证。 |
| **必须建成的能力** | • 支持围绕某日、某周、某 repo、某 topic、某趋势簇进行追问。  <br>• 回答必须可追溯到事件、主题、时间线与原始对象。  <br>• 支持典型分析动作：变化解释、时间窗口比较、推荐阅读、主题推进总结。  <br>• 形成“日报是入口，问答是深挖”的消费路径。 |
| **阶段产出** | • Ask 页面 / 对话入口  <br>• 基于历史摘要与证据链的问答引擎  <br>• 高价值查询模板 |
| **退出标准** | 用户开始把它当作‘技术分析助手’，而不是只会推送日报的系统。 |

# M8｜个性化与信号学习

|     |     |
| --- | --- |
| **目标状态** | 让同一套情报底座对不同用户输出不同价值密度：排序、摘要、推荐、提醒开始真正因人而异。 |
| **必须建成的能力** | • 支持 watch 优先级、方向偏好、语言/领域偏好、摘要粒度偏好。  <br>• 收集并学习用户的点击、打开、标记“值得/不值得”、继续追问等反馈。  <br>• 对日报排序、推荐阅读、提醒阈值、趋势解释进行个性化调整。  <br>• 形成用户级画像与偏好模型。 |
| **阶段产出** | • Feedback Loop  <br>• Preference Profile  <br>• 个性化排序与推荐策略 |
| **退出标准** | 两个用户看到的首页、日报重点和推荐对象开始明显不同；系统从统一产品变成个人技术秘书。 |

# M9｜协作与行动闭环

|     |     |
| --- | --- |
| **目标状态** | 把个人使用能力提升为团队基础设施：团队可以共享 watch、共享结论、共享关注对象，并把情报转成后续动作。 |
| **必须建成的能力** | • 支持 Team Space、共享 watchlist、共享 topic、团队模板。  <br>• 支持在情报条目上批注、收藏、标记负责人与 follow-up。  <br>• 支持告警策略、重点对象升级通知、周会简报生成。  <br>• 让高价值信号转化为团队动作，而不是停留在阅读层。 |
| **阶段产出** | • 协作空间与权限模型  <br>• 批注/标记/指派/告警模块  <br>• 团队简报与共享消费路径 |
| **退出标准** | 产品从个人助手成长为团队情报基础设施；其价值开始体现为团队协同效率与共同认知的提升。 |

# M10｜平台化与最终态发布

|     |     |
| --- | --- |
| **目标状态** | 将产品升级为成熟平台：拥有开放集成能力、企业级治理能力、稳定商业交付能力，并对外形成明确终局形态。 |
| **必须建成的能力** | • 开放 API、Webhook / GitHub App 模式、外部消息平台集成。  <br>• 支持组织级治理：角色权限、审计、配额、计费、合规、安全策略。  <br>• 构建高可用架构、回溯重算、模型质量评估与运营后台。  <br>• 明确商业化包装：个人版、专业版、团队版、企业版。 |
| **阶段产出** | • 平台 API 与连接器框架  <br>• 企业后台与治理能力  <br>• 稳定可发布、可销售、可运营的产品版本 |
| **退出标准** | 系统具备成熟 SaaS / 平台产品特征，可以面向真实个人用户、团队与企业持续交付，并具备长期演进空间。 |

# 三、推荐使用方式

• 这份文档适合作为长期 Roadmap，而不是当前迭代 Backlog。后续应始终以“当前处于哪个里程碑、距离下一个状态差什么”为切入点继续细分。

• 后续继续拆解时，建议只拆相邻里程碑之间的跨越，例如：M2→M3、M4→M5、M8→M9，而不要一次性把所有阶段细成任务。

• 每个里程碑的细分都应同时覆盖四条线：数据与算法、产品与交付、体验与增长、平台与治理。

# 四、最终态一句话定义

**最终态产品不是“GitHub 事件摘要器”，而是：**一个面向个人与团队的 GitHub 技术情报平台，能够持续监控仓库、主题与生态趋势，生成多周期报告，支持证据链追问、个性化优先级、协作与行动闭环，并以平台化与企业化能力对外稳定交付。

# 五、建议优先拆解的跨越

|     |     |     |
| --- | --- | --- |
| **跨越区间** | **最值得先细拆的主题** | **原因** |
| M1 → M2 | 统一 schema、normalizer、entity/topic resolver、去重压缩 | 这是后续所有摘要、推荐、问答与主题判断的共同底座。 |
| M2 → M3 | Repo 摘要、重要性评分、推荐阅读、Today 首屏 | 这是产品第一次出现可消费价值的节点。 |
| M3 → M4 | TopicWatch、Topic Timeline、证据链与提案推进判断 | 这是产品差异化最强的一跳。 |
| M6 → M7 | History-grounded 问答、比较查询、追问模板 | 这是从阅读产品变成分析助手的一跳。 |
| M8 → M9 | Team Space、共享 watch、批注与 follow-up | 这是从个人工具升级为团队基础设施的一跳。 |
