# GitHub 监控聚合 Agent PRD

## 1. 基本信息

**产品名称（暂定）**  
GitHub Intel Agent

**版本**  
PRD v1.0

**产品类型**  
面向开发者的 GitHub 情报聚合与摘要 Agent

**一句话定义**  
持续跟踪仓库、主题、趋势榜单与跨仓库议题，把分散的 GitHub 弱信号压缩为每日可读、可追问、可回溯的高价值技术情报。

---

## 2. 背景与问题定义

### 2.1 背景

重度开发者、开源贡献者、技术研究者、技术管理者，往往会同时关注多个 GitHub 仓库、组织、技术提案与生态趋势。

典型问题：

1. **仓库太多，事件太碎**
   - 如 Node.js、React、Rust、TC39 等，日常有大量 commit、PR、issue、release
   - 手动逐个看成本极高

2. **关注对象不总是单个仓库**
   - 比如 TC39 提案往往分散在多个仓库
   - 用户真正关心的是“这个提案/方向有没有推进”，而不是某个仓库今天发生了几个动作

3. **趋势来源多样且异构**
   - GitHub Trending 是日榜快照
   - github-weekly-rank 是周聚合结果
   - 它们不是标准 repo event stream

4. **现有 GitHub 通知过于“事件导向”**
   - 用户收到的是大量原子事件
   - 缺少跨仓库聚合、重要性排序、主题归并、与昨日/上周 diff、个性化摘要

### 2.2 核心问题

用户并不需要“更多 GitHub 通知”，而需要：

- 今天**最值得关注**的变化是什么
- 某个关注方向或提案是否在推进
- 与昨天相比，什么在升温，什么在降温
- 哪几个 PR、issue、repo 值得花时间点进去看
- 哪些看似分散的事件，其实是在描述同一件事

---

## 3. 产品目标

### 3.1 产品目标

构建一个 GitHub 情报 Agent，帮助用户：

1. **每日低成本了解重点变化**
2. **按主题而不是按仓库理解 GitHub 动态**
3. **快速发现值得深看的 PR、issue、repo**
4. **跟踪趋势变化，而不仅是看静态榜单**
5. **把日报变成可追问、可回溯的长期情报记忆**

### 3.2 非目标

本产品第一阶段**不做**：

- 代码托管替代品
- 全功能 GitHub 客户端
- 通用告警平台
- 企业级安全审计系统
- 全自动深度代码审查
- 所有 GitHub 公开内容的全文搜索引擎

- 可追问的历史记忆

## 5. 核心使用场景

### 场景 1：多个仓库的每日监控摘要

用户关注 Node.js、React、Vite、Rust 等多个 repo，希望每天上午收到一份摘要：
- 今日最重要变化
- 值得阅读的 PR
- 热 issue
- 新 release
- 与昨日相比的变化

### 场景 2：TC39 提案跟踪

用户关注某个 TC39 proposal，希望系统按主题归并：
- proposal 仓库更新
- notes 中是否提到
- agenda 是否进入会议
- ecma262 / test262 是否出现联动

最终输出：
- “这个 proposal 今日是否有推进”
- “推进是在讨论层面还是工程层面”

### 场景 3：GitHub Trending 的每日 diff

用户希望每天看到：
- 今日新上榜仓库
- 掉榜仓库
- 排名上升 / 下降
- 热门方向聚类
- 与昨日相比的结构性变化

### 场景 4：周榜聚合

用户希望结合 `github-weekly-rank` 等榜单，获得：
- 周级热度信号
- 日榜与周榜是否共振
- 哪些项目是短时噪声，哪些是强趋势

### 场景 5：日报后的可追问

用户看完日报后继续问：
- 今天最值得看的 3 个 PR 是什么
- Temporal 这周有无实质推进
- 今天 Trending 里哪些项目和 Agent 相关
- Node.js 这周比上周活跃在哪些模块

## 6. 产品方案概述

产品核心采用如下抽象：

**Watch Target -> Signals -> Canonical Events -> Topic Aggregation -> Narrative Report**

即：
1. 用户定义关注对象
2. 系统采集原始信号
3. 转换为统一事件语义
4. 按主题与价值进行聚合
5. 生成日报、周报与可追问记忆

---

## 7. 核心概念设计

### 7.1 Watch Target

Watch Target 是产品最重要的对象，不只是 repo。

支持四类：

#### A. RepoWatch
用于关注单个或多个仓库  
例：
- nodejs/node
- facebook/react

#### B. TopicWatch
用于关注一个主题或跨仓库议题  
例：
- TC39 Temporal
- TC39 Async Context
- Node Permission Model
- AI Coding Agent

#### C. TrendWatch
用于关注趋势榜单  
例：
- GitHub Trending 日榜
- GitHub Trending JavaScript
- GitHub Trending AI 方向

#### D. RankFeedWatch
用于关注外部榜单源  
例：
- github-weekly-rank

### 7.2 信号类型

系统输入为多种异构原始信号，包括：

- GitHub repo events
- PR / issue / commit / release 详情
- 榜单快照
- meeting notes
- agenda 变更
- 主题仓库文档变更

### 7.3 统一事件语义

系统内部统一为 Canonical Event，例如：

- PR Opened
- PR Merged
- Issue Opened
- Issue Became Hot
- Release Published
- Repo Entered Trending
- Repo Left Trending
- Rank Moved Up
- Proposal Discussed
- Proposal Integration Signal
- Proposal Testing Signal

## 8. 功能需求

### 8.1 Watch 配置

#### 功能描述
用户可创建和管理不同类型的 Watch Target。

#### 子功能
1. 添加 RepoWatch
2. 添加 TopicWatch
3. 添加 TrendWatch
4. 添加 RankFeedWatch
5. 设置优先级
6. 设置通知频率
7. 设置摘要粒度

#### 用户故事
- 作为用户，希望关注多个 repo，而不是一次只能看一个
- 作为用户，希望直接关注“TC39 Temporal”，而不是手动配置多个相关仓库
- 作为用户，希望能把某些对象设为高优先级

#### 验收标准
- 用户可新增、编辑、删除 Watch Target
- 每个 Watch Target 可设置摘要频率和优先级
- TopicWatch 支持绑定多个 repo、关键词、规则

### 8.2 日报生成

#### 功能描述
系统每天定时生成一份摘要报告。

#### 报告结构
1. 今日最重要变化
2. 按 Watch Target 分类摘要
3. 推荐阅读 PR / issue / repo
4. 趋势变化
5. 与昨日 diff
6. 需要关注的异常 / 突增 / 突降

#### 用户故事
- 作为用户，希望每天 5 分钟内看完重要变化
- 作为用户，希望日报不是流水账，而是重点排序后的摘要

#### 验收标准
- 每日报告可读时长控制在 3~8 分钟
- 报告中每个 section 都有明确标题与重点
- 至少提供“今天最值得关注的 3 件事”

### 8.3 Topic 聚合

#### 功能描述
系统把跨仓库信号归并为主题级摘要。

#### 子功能
1. Topic 配置规则
2. 关键词 / 仓库 / 别名映射
3. 主题信号聚合
4. 主题推进判断

#### 用户故事
- 作为用户，希望系统告诉我某个 proposal 有没有推进
- 作为用户，希望多个 repo 的事件最后归结成一个主题判断

#### 验收标准
- 一个 TopicWatch 可绑定多个 repo 与匹配规则
- 系统可将多条事件聚合为一个主题摘要
- 输出内容以“主题”为主语，而不是以“仓库”为主语

### 8.4 Trending Diff

#### 功能描述
系统每天采集 Trending 快照，并生成相对昨日的变化摘要。

#### 子功能
1. 今日榜单快照
2. 昨日快照对比
3. 新上榜
4. 掉榜
5. 排名上升 / 下降
6. 方向聚类

#### 用户故事
- 作为用户，希望知道今天哪些仓库第一次出现
- 作为用户，希望知道今天的热榜与昨天相比有何结构性变化

#### 验收标准
- 系统每天可展示新上榜 / 掉榜 / 排名变化
- 对榜单仓库进行主题聚类
- 输出 1~3 条趋势结论

### 8.5 Weekly Rank 结合分析

#### 功能描述
系统引入外部周榜数据，为日报 / 周报提供宏观背景。

#### 子功能
1. 周榜采集
2. 与 Trending 对照
3. 短期热度与中期热度交叉判断

#### 用户故事
- 作为用户，希望分辨今天爆火的是短时热点还是持续趋势

#### 验收标准
- 对于出现在日榜与周榜中的项目，系统给出“强趋势”标记
- 对只出现在日榜而未出现在周榜中的项目，给出“短时热点”倾向标记

### 8.6 推荐阅读

#### 功能描述
系统从大量变化中筛出“最值得点进去看”的对象。

#### 推荐对象
- PR
- Issue
- Repo
- Release

#### 筛选逻辑
综合考虑：
- 事件重要性
- 影响范围
- 讨论热度
- 用户兴趣贴合度

#### 用户故事
- 作为用户，希望不是看完摘要就结束，而是知道接下来该点哪几个链接

#### 验收标准
- 每日报告中至少推荐 3 个值得深读对象
- 推荐理由清晰可解释

### 8.7 历史回溯与可追问

#### 功能描述
所有报告与关键事件可长期保存，并允许用户继续追问。

#### 追问能力示例
- “这个 proposal 最近 7 天有哪几次关键推进”
- “Node.js 这周最重要的 5 个 PR”
- “今天 Trending 里的 AI Agent 项目有哪些共性”

#### 用户故事
- 作为用户，希望日报不是一次性消费，而是知识积累入口

#### 验收标准
- 系统保留历史报告
- 用户可基于历史报告继续追问
- 每条结论都尽量可追溯到原始对象

## 10. 核心交互流程

### 10.1 首次配置流程

1. 用户进入产品
2. 选择感兴趣对象：
   - 仓库
   - 主题
   - 趋势源
3. 设置摘要频率：
   - 日报
   - 周报
4. 设置重点领域：
   - JS / Runtime / AI / Infra 等
5. 完成初始化
6. 次日开始接收情报摘要

### 10.2 每日报告阅读流程

1. 用户打开日报
2. 查看“今天最值得关注的 3 件事”
3. 查看高优先级 Watch Target 的摘要
4. 查看推荐阅读对象
5. 点进具体 PR / repo / topic
6. 对条目做反馈：
   - 值得
   - 不值得
   - 多推荐这类
   - 少推荐这类

### 10.3 Topic 深挖流程

1. 用户进入 Topic 页面
2. 选择如 “TC39 Temporal”
3. 查看：
   - 今日 / 本周摘要
   - 时间线
   - 关键证据
   - 相关 repo / PR / notes / agendas
4. 继续提问

## 12. 核心排序逻辑

### 12.1 重要性评分

每条候选对象计算 `importance_score`

评分维度：
1. **事件权重**
   - release > merged PR > opened PR > issue > push
2. **影响范围**
   - 改动文件数
   - 是否核心目录
   - 是否 API / breaking
3. **讨论热度**
   - 评论数
   - review 数
   - participant 数
4. **用户兴趣贴合**
   - 是否高优先级 watch
   - 是否用户常读方向
5. **去噪权重**
   - bot 降权
   - typo / docs-only 降权
   - mass update 降权

### 12.2 聚合逻辑

#### Repo 摘要聚合
把 repo 内大量事件压缩成少量结论：
- 今日最重要变更
- 热点 PR
- 热点 issue
- activity delta

#### Topic 摘要聚合
把跨仓库事件整合为：
- 讨论推进
- 规范推进
- 测试推进
- 阶段变化候选信号

#### Trending 聚合
把榜单变化整合为：
- 新热点
- 持续热点
- 下行热点
- 方向性变化

定义：
- 打开日报

- 点击至少一个推荐对象

### 14.2 核心指标

#### 使用指标
- 日报打开率
- 周活跃用户数
- Watch Target 平均数量
- 用户平均阅读时长
- 日报后续点击率

#### 质量指标
- 用户标记“值得”的比例

## 17. 风险与挑战

### 17.1 噪声问题

风险：
- 事件很多，但用户不觉得有价值

应对：
- 强打分 + 去重 + 主题聚合
- 报告必须有“最重要 3 件事”

### 17.2 Topic 识别准确度

风险：
- 跨仓库主题归并不准

应对：
- 第一阶段采用人工规则 + 白名单
- 高频 topic 逐步沉淀模板

### 17.3 趋势信号不稳定

风险：
- Trending 天然波动大

应对：
- 用昨日 diff + 周榜交叉验证
- 将日榜与周榜区分为不同强度信号

### 17.4 用户首次配置成本

风险：
- 用户不知道该关注什么

应对：
- 提供模板：
  - Node.js 观察包
  - TC39 观察包
  - AI Agent 观察包
  - Web Infra 观察包

## 19. 产品成功标准

产品成功不是“抓到了多少事件”，而是：

1. 用户真的愿意**每天看**
2. 用户觉得它**替代了一部分手动刷 GitHub 的时间**
3. 用户能从摘要里找到**值得深读的对象**
4. 用户开始把它当成一个**长期技术情报助手**

---

## 20. 最终产品定位

这个产品不是一个“GitHub 事件通知器”。

它应该被定义为：

> 一个面向开发者和技术观察者的 GitHub 情报 Agent，  
> 将仓库变化、跨仓库议题、热榜波动与周级趋势聚合成可读、可追问、可积累的技术情报。
